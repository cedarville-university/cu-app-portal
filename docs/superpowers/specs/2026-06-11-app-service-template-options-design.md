# App Service Template Options Design

## Purpose

The portal should support multiple Azure App Service starter templates while keeping the create flow easy for non-specialist users to choose from. The first implementation remains App Service-only and expands the current single Next.js starter into a small catalog of runtime-oriented templates with clear descriptions, database choices, and Microsoft Entra login choices.

## Goals

- Add a template model that can describe Azure App Service runtime details for Node, Python, and Java starters.
- Let users choose whether to include a database and Entra login when a selected template supports those features.
- Preserve the existing create-to-download, managed repository, and one-step Azure publish behavior for the current Next.js template.
- Add template descriptions that explain why a user would choose each template, not just what technology it uses.
- Keep unsupported feature combinations out of the UI and validation path.

## Non-Goals

- No Azure Functions, Static Web Apps, Container Apps, Kubernetes, or custom container support in this milestone.
- No dynamic discovery of Azure runtime stacks from Azure CLI during normal portal use.
- No broad redesign of My Apps, download pages, repository import, or deletion flows.
- No production support for every possible framework in each runtime family.

## Recommended Initial Catalog

The first catalog should include these templates:

- **Next.js Web App**: Use this for full web applications with pages, forms, server-side logic, and Cedarville-styled UI. It is the best default when an app needs both frontend and backend behavior in one project.
- **Express API**: Use this for lightweight JavaScript APIs, webhooks, integrations, and backend services that do not need a full page-based UI.
- **Python FastAPI**: Use this for Python-backed APIs, data workflows, automation endpoints, or apps that benefit from Python libraries while still deploying cleanly to Azure App Service.
- **Java Spring Boot**: Use this for larger service-oriented applications, enterprise Java patterns, dependency injection, and teams already comfortable with Spring.

The Next.js template should be converted first and remain behaviorally equivalent to the current `web-app` starter. FastAPI should be the first new runtime template because it proves non-Node App Service generation with a compact starter. Express and Spring Boot can follow after the runtime-aware publishing path is proven.

## Template Metadata

Each template should declare both user-facing metadata and machine-facing capabilities.

The user-facing fields should include:

- `name`
- `description`
- `decisionSummary`, a short sentence shown on the template card to help users decide
- `bestFor`, a short list of use cases

The runtime metadata should include:

- `family`: `node`, `python`, or `java`
- `framework`: for example `nextjs`, `express`, `fastapi`, or `spring-boot`
- `displayName`: for example `Node.js 24 / Next.js`
- `azureRuntimeStack`: for example `NODE|24-lts`, `PYTHON|3.14`, or a Java App Service stack
- `startupCommand`: the command Azure App Service should use when required
- `workflowFileName`: the GitHub Actions workflow file generated by the template

The feature metadata should include:

- `database`: `unsupported`, `optional`, or `required`
- `databaseProviderOptions`: initially empty or `["PostgreSQL"]`
- `entraLogin`: `unsupported`, `optional`, or `required`

The create form should render database and Entra controls only when a feature is optional. Required features should be submitted as hidden inputs. Unsupported features should not be submitted as enabled choices.

## Create Input Model

The create action should accept a normalized input shape:

- `templateSlug`
- `appName`
- `description`
- `hostingTarget`, fixed to `Azure App Service` for this milestone
- `databaseProvider`, either `none` or `postgresql`
- `entraLogin`, boolean

Validation must be template-aware. A request is valid only when:

- the selected template exists and is active
- the hosting target is Azure App Service
- `databaseProvider` matches the template database capability
- `entraLogin` matches the template Entra capability
- the existing app name and description rules still pass

This validation should live near the existing create-app schema, but it should receive the selected template or a normalized capability object instead of only a list of hosting targets.

## Generation Design

Template generation should continue to render `templates/<slug>/files` and generated files through `buildSourceSnapshot`. The snapshot builder should become capability-aware:

- Render base files for every template.
- Include database files only when the input uses PostgreSQL.
- Include Entra/auth files only when the input enables Entra login.
- Emit a runtime-aware deployment manifest for every App Service template.

For the first step, the current Next.js template can keep all current files while the input defaults to PostgreSQL and Entra enabled. After parity is preserved, optional variants can conditionally omit Prisma/database files and NextAuth/Entra files.

Generated code should stay idiomatic for each runtime. Avoid a single mega-template with many language-level conditionals. Shared portal-generated files such as `app-portal/deployment-manifest.json`, deployment docs, and publish skills can remain generated centrally when their content is derived from the same manifest.

## Deployment Manifest

The deployment manifest should describe the generated app instead of assuming Node/Next/PostgreSQL/Entra.

It should include:

- runtime family, framework, version/display label, Azure runtime stack, and startup command
- hosting provider and service, fixed to Azure App Service
- GitHub Actions workflow metadata
- database section only when PostgreSQL is selected
- auth section only when Entra login is selected
- application settings required by the selected features

Existing consumers should be updated to read these values rather than hardcoding `NODE|24-lts`, `npm start`, `DATABASE_URL`, and Entra settings.

## Azure Publishing

The Azure publish runtime should remain the only production publish runtime for generated templates in this milestone. It should become conditional:

- Always create the Azure Web App in the configured shared resource group and App Service Plan.
- Use the template runtime stack and startup command.
- Create a per-app PostgreSQL database only when the request selected PostgreSQL.
- Set `DATABASE_URL` only when a database exists.
- Configure Entra redirect URI and auth app settings only when Entra login is enabled.
- Continue setting GitHub OIDC and workflow secrets required for deployment.

The portal should keep one-step create-and-publish enabled only for templates whose runtime and workflow are supported by the publishing provider.

## UI Design

The create template list should help the user choose the right template. Template cards should show:

- template name
- decision-focused summary
- runtime/framework label
- short best-for hints
- badges for database and Entra support

The selected template page should show the same decision context above the form. The form should keep the current app name and description fields, then show concise database and login choices when applicable.

The UI should avoid exposing Azure implementation details as primary decision text. Runtime labels are useful; App Service stack strings are not.

## Testing

Unit coverage should include:

- catalog serialization for new runtime and capability metadata
- create validation for supported and rejected database/auth combinations
- form rendering for unsupported, optional, and required feature controls
- deployment manifest output for database/auth on and off
- source snapshot generation for at least Next.js parity and one new runtime template
- Azure publish runtime behavior with and without database and Entra

End-to-end coverage should keep the existing create-and-download path green for the Next.js template. A second focused e2e or integration test can verify selecting a new template generates a downloadable artifact without publishing.

## Rollout

Implement in phases:

1. Add metadata, validation, and UI support while preserving current Next.js behavior.
2. Convert the current template into the new capability model.
3. Add optional database and Entra handling for Next.js.
4. Add the FastAPI template and prove archive/repository generation.
5. Add Express and Spring Boot templates once the new template path is stable.
6. Update setup and template-authoring docs with the App Service-only model and template-authoring expectations.
